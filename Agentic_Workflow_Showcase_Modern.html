01. 最先进的 AI Coding 架构演进

为了理解为什么我们需要工作流，必须先认清目前行业处于什么阶段，以及单体大模型（LLM）的瓶颈在哪里。技术演进正在经历三个截然不同的阶段：

GEN 1
Copilot 时代
高级打字机

基于上下文的单点补全。 IDE 插件截取光标前后的短文本，发送给模型。它只能解决“怎么写这个函数”的语法问题，完全缺乏项目全局视图，无法解决架构问题。

GEN 2
Chat & RAG 时代
带搜索引擎的外包

基于检索的问答与单步修改。 通过向量搜索将相关文件拼接到 Prompt。但它缺乏纪律，收到指令就立刻动工，容易破坏分层架构。而且多轮对话后会导致严重的“失忆”与上下文污染。

GEN 3 (We are here)
Agentic Workflow
有纪律的系统工程师

引入状态机、多智能体与工具链。 模型不再是直接的代码生成器，而是被嵌入到一个有严格规则控制的工作流中。人类只做高层架构决策，AI 负责底层的脏活累活并强制闭环验证。

核心论点： 单体大模型解决不了复杂的软件工程问题，我们需要的不是更聪明的大模型，而是“控制大模型行为的架构协议”。

02. 解构三剑客：Agent / Rules / Skills

很多开发者对 AI 概念混淆不清。在我们的架构中，必须用 软件工程的语言 重新定义它们，消除抽象感。

The Core Primitives Architecture

Rules (规则)

静态宪法

被动触发的、声明式（Declarative）的约束条件（如 `.mdc` 文件）。

作用：定义系统的边界与红线。
类比：项目中的 Linter 和团队架构规范手册。

Main Agent

Orchestrator

作为网关和项目经理，负责接受 Rules 监督，拆解任务，分配执行。

Skills (技能)

动态 SOP

主动加载的、命令式（Imperative）的执行手册（如 TDD）。

作用：赋予 AI 特定领域的专业方法论。
类比：某项业务的标准化操作指南。

Subagents (并发执行集群 / 微服务)

Subagent A
探索前端架构
Subagent B
分析后端 API
Subagent C
编写测试用例

由主代理派生出，具有独立上下文。实现**分而治之，隔离污染**，防止长文本导致的记忆混乱。

04. 源码级深剖：核心规则逐行拆解

听众可能会好奇：你凭什么能“卡死”不可控的大模型？我们不谈虚的，直接翻开这两份“圣经”的源码，一步一步拆解黑魔法是如何通过改变底层 Logits 分布来生效的。

Engine Alpha

The Gatekeeper：“守门人”引擎

FILE: new-task-kickoff.mdc

大模型有一种天生的“生成冲动”（Generative Urge），看到需求就忍不住敲代码。守门人的唯一职责，就是在代码生成的入口处布下四道关卡，强行将其从“生成态”打断为“挂起态”。

1 强制上下文嗅探 (Phase 1)

当接收到新任务，AI 绝对不允许立刻作答。源码要求它必须在后台静默调用系统命令，执行六重扫描：

1.0 嗅探 `plugin_hook` 提取架构配置
1.1 检索历史对话 `agent-transcripts/`
1.2 匹配系统核心设计文档
1.4 分析涉及的核心源代码模块
1.5 `git log --oneline -20` 近期变更
1.6 提取双层记忆教训 `lessons.md`

注：特别是 1.6 步。昨天排查踩的坑写进 lessons，今天全团队的 AI 都会自动避开，形成“自我进化的群体大脑”。

4 智能路由分发与执行链 (Phase 4)

当人类下达“通行证”（如说“选B，直接写计划”），守门人立刻放行，并动态路由：

  • ➜ 路径 A： 唤起 brainstorming 进行深度设计。
  • ➜ 路径 B： 唤起 writing-plans 拆解任务，然后切入 TDD 编码。
  • ➜ 路径 D： 唤起 systematic-debugging 走四阶排错流程。
模型成本调度策略： 源码中定义了严格的层级。设计与审查任务强制使用 Opus（T1）；大量代码生成使用 Codex（T2）；机械的搜索与 Git 动作，强制降级派发给低成本的 Fast 模型（T4）静默执行。
Engine Beta

The Constitution：“钢铁纪律”引擎

FILE: ai-coding-protocol.mdc

如果说守门人负责管入口，那么最高宪法则负责管全程。它是 alwaysApply: true 的全局规则，如同紧箍咒一样套在主 Agent 头上，防范其在漫长链路中出现幻觉、过度工程或代码污染。

The Auto-Init Sequence

1. 自动初始化流程：代码编写前的 5 步强制挂载

大模型决不能“凭空猜测”项目规范。最高宪法规定，在每次发生代码编写前，AI 必须在后台暗中执行一套不可跳过的 Read-Before-Write 自动初始化流程，确保代码风格 100% 无缝融入原仓库：

Step 1: Check 优先嗅探是否已有 learned-conventions 基线字典
Step 2: Sample 若无字典，强制暗中读取同目录 2-3 个真实文件
Step 3: Extract 现场提炼错误处理、日志习惯与类型密度
Step 4: Align 按出现频率覆盖 AI 自带的“大众风格”
Step 5: Verify 动笔后一致性校验，修正一切突兀语法

2. 最小有效变更 (Smallest Effective Diff)

大模型喜欢“借题发挥”。宪法严令：“偏好能解决请求的最小改动，除非被明确要求，绝对不要重构无关代码”。

3. 严禁盲造轮子 (Check Before Creating)

在创建任何新的抽象类或工具文件前，必须强制全库检索有没有现成的 Utilities 或 Helper 可以复用。

4. 交卷前自查 (Diff Self-Review)

在告诉用户“任务完成了”之前，AI 必须暗中运行命令检查自己的 git diff，清理多余的换行、冗余的 Import 或临时日志，最终才能交付干净的代码。

05. 四阶段状态机运转 (FSM)
