# Design-Impl Audit — 设计 vs 实现 对账副驾驶

## 这个 skill 是什么（自说明）

**一句话**：研发一手拿设计文档（意图真相）、一手拿 Git 提交（交付真相），本 skill 是"**副驾驶**"——**不替研发改代码，只替研发把关**。开工前帮你列"这次要落哪些点"（**提醒**），提交后帮你核对"到底落了哪些、落偏了哪些"（**检查**）。skill 的产出物是"**清单 + 报告**"，不是 patch。

**解决的痛点**：
- 设计文档会持续演进，一次迭代可能涉及跨阶段、跨模块 10+ 处变更，人脑记不全；
- 研发写代码时容易漏掉"文档角落里那一句配置要求"或"某个交互分支"；
- 提交评审只看 diff 局部，难以回答"对着蓝图我们到底做到了百分之几"；
- 文档和代码不同步会让下一次迭代失去基准，技术债复利滚雪球。

**定位**：不是替研发做决定，而是把"设计 → 代码"这条链路上**每个本该被核对的点**都暴露出来，让人一眼看清差距。

**适用范围**：任何语言、任何规模的代码库，只要同时有"Markdown/HTML/文本格式的设计文档"和"Git 版本控制"。

## 安装（零代码，拷贝即用）

本 skill 就是**一份 SKILL.md 文件**，把它放进你所用编辑器/Agent 的 skill 目录即可生效。

### 前置准备

- 目标目录下新建一个子文件夹（建议名 `design-impl-audit`）
- 把本 `SKILL.md` 拷进去，完成

### 各编辑器/Agent 安装路径

| 工具 | 全局（所有项目可用） | 项目内（仅当前项目可用） |
|---|---|---|
| **Cursor** | `~/.cursor/skills/design-impl-audit/SKILL.md` | `<project>/.cursor/skills/design-impl-audit/SKILL.md` |
| **Claude Code** | `~/.claude/skills/design-impl-audit/SKILL.md` | `<project>/.claude/skills/design-impl-audit/SKILL.md` |
| **Codex CLI** | `~/.codex/skills/design-impl-audit/SKILL.md` | `<project>/.codex/skills/design-impl-audit/SKILL.md` |
| **其他支持 Agent Skills 的工具** | 按该工具的 skill 目录约定放置 | 同上 |

### 验证安装

安装后，向 AI 说以下任一触发词即可调用：
- "帮我做一次设计对账"
- "用设计实现审计 skill 审一下这批 commit"
- "design audit this commit range"
- "检查 xxx 文档和最近提交的代码一致性"

AI 会自动读取本 SKILL.md 并按内部流程执行。

### 使用依赖

- 本 skill 依赖 AI 能读取本地文件（读文档）、能执行 `git` 命令（拿 commit 信息）、能抓取 URL（远端文档）。主流 Agent 工具（Cursor / Claude Code / Codex 等）默认都具备这些能力，**无需额外配置**。
- 如果设计文档在 GitLab/GitHub 私有仓库且需要认证，需把文档先拉到本地再给本 skill，或让 AI 用已登录的 `gh`/`glab` CLI 抓取。

## 怎么用（调用方式）

### 场景 1：研发说"给我这次迭代的落地计划"
→ 输入：设计文档（刚定稿或刚更新）。
→ 本 skill 走 §流程 Step 2 把文档拆成**验收条目清单**（带阶段/意图/参数/交互四维），**不跑 Git**，直接给研发一份"本次要落的点一览"。研发据此拆任务、分 commit。

### 场景 2：研发说"我这批 commit 已经改完，帮我审一下"
→ 输入：Git 提交范围 + 设计文档。
→ 本 skill 走 §流程 Step 1→5 全链路，产出**对账报告**。每条验收项落成 ✅/⚠️/❌/➕ 四档之一，偏差项附一句最小修复方向。

### 场景 3：研发说"合并前我想确认一下没漏没偏"
→ 输入：分支 vs 目标分支 + 设计文档。
→ 同场景 2，但额外强调**兜底检查**：对 ❌ 未实现项做"二次搜索"（扩大 git log 范围或搜同名符号），排除"已经在别的 commit 做完了"的误报，把真实风险暴露给用户。

### 场景 4：研发说"文档和代码哪个老了，帮我对一下"
→ 输入：设计文档 + 当前分支状态（或一段时间窗）。
→ 本 skill 做**双向扫描**：
- 正向：文档每条 → 代码有没有对应实现？（同场景 2）
- 反向：代码里的关键新增（新函数、新配置项、新接口参数）→ 文档有没有写？
→ 输出**差异矩阵**，分两栏："代码需补齐文档" / "文档需补齐代码"，供研发决策是改代码还是改文档。

## 运行时输入（能自判就不要问用户）

研发触发本 skill 后，skill **先自行从上下文/文档/Git 推断能推的部分，只把实在推不出或有歧义的点问用户**。禁止把用户已经明确给出的信息再问一遍。

### 需要的三项输入

**Q1. 代码范围**（场景 1 除外）：
- 合法形式：Commit 范围 `A..B` / 单个 commit hash / 分支对比 / 时间段+作者 / 指定文件列表
- 模糊描述处理：用 `git log --oneline --author=... -30` 搜候选让用户挑

**Q2. 设计文档**：
- 本地路径（md / html / txt / docx 等）
- HTTP/HTTPS URL（skill 通过 WebFetch 抓取）
- 需要认证的私有仓库文档：让用户先拉到本地再给路径

**Q3. 可选聚焦**：
- 是否只关注某个模块/目录（缩小扫描面）
- 报告落盘路径（默认：项目根下 `docs/audit-report/YYYY-MM-DD-<文档名>-设计实现对账.md`，若该目录不存在则 fallback 到 `<设计文档所在目录>/audit-YYYY-MM-DD-<文档名>.md`）

### 自动裁决原则（优先级从高到低）

1. **用户明说的优先**：凡是用户在当前或最近一条消息里明确给出的，直接用，不再问。
2. **设计文档自带分阶段标尺**：若文档有明确的分阶段标记（如 P1/P2、阶段一/阶段二、MVP/增强、v1/v2、Phase A/B 等），优先取"时间上对应本次 commit 的那一段"作为正向标尺，其他段做**负向越权校验**。判断依据是 commit 时间 vs 文档里的交付节奏。
3. **文件清单自动过滤**：从 `git log --stat` 拿到的文件列表，自动剔除噪声（测试数据产物、二进制文件 `*.class/*.dex/*.so/*.pyc` 等、构建产物 `build/ dist/ target/ node_modules/`、lockfile、IDE 配置、一次性脚本），无需问用户。
4. **报告路径约定默认**：见上文 Q3。
5. **仅在真正不确定时才问**：例如设计文档路径无法从上下文推断、或有多份候选文档时。此时用一次性 AskQuestion 合并问清，不要分多轮。

### 自动裁决后的复述

进 Step 1 之前，**用一段话把自动裁决结果复述给用户**（"标尺=<正向阶段> 全部条目、<其他阶段> 做负向校验；文件清单 N 个已过滤噪声；报告落盘至 xxx"），让用户一眼看到有没有错，错了当场叫停。这条**替代**了传统的强制确认闸门——能推就推，推错了用户会喊停。

## 内部流程（5 步，严格顺序）

### Step 1. 锁定代码范围（场景 2/3/4 必做，场景 1 跳过）
- `git log --stat <范围>` + `git diff --stat <范围>` 拿到改动文件清单 + 行数。
- 自动过滤噪声（见上节"自动裁决原则"第 3 条）。
- **把过滤后的结果作为"自动裁决复述"的一部分打给用户**，一段话讲清。不再单独设闸门；用户有意见就当场喊停，没意见就继续 Step 2。

### Step 2. 解析设计文档 → 验收条目清单 + 标尺自动裁决
按文档一级/二级标题切分，每条目记录：

| 字段 | 说明 |
|---|---|
| `id` | 章节号或自编序号（如 §X.Y、D1-1、F-03 等，遵循文档已有编号系统） |
| `phase` | 所属阶段（若文档有分阶段；否则为空） |
| `intent` | 一句话概括设计意图 |
| `expectations` | 具体期望（行为 / 参数 / 接口 / 交互），分点列 |
| `doc_ref` | 文档原文定位（锚点或行号） |

**标尺自动裁决**：
- 若文档有明确分阶段：按 Step 1 的 commit 时间 × 文档分阶段节奏，自动选"正向标尺=本次应该完成的那一段"，其他段自动转入**负向越权校验**（出现了不该出现的能力算 ⚠️）。
- 若文档无分阶段：全部条目都是正向标尺。
- 裁决结果写进给用户的复述里，一段话讲清，不单独设确认闸门。

场景 1 到此结束，把这份清单交付即可。

### Step 3. 建立代码证据库
对 Step 1 确认的每个文件 diff 做**语义摘要**（严禁贴原文代码），每条记录：
- 文件路径 + 函数/类名
- 本次改了什么（新增函数 / 修改分支 / 调参数 / 加配置项）
- 关键参数或阈值的新旧值

### Step 4. 逐条对账
对 Step 2 每条验收条目，去 Step 3 证据库里找实现：

| 状态 | 判定标准 | 报告处理 |
|---|---|---|
| ✅ **完全实现** | 有明确代码对应 + 参数/行为与文档一致 | 引用代码位置即可 |
| ⚠️ **有偏差** | 有代码但参数/边界/交互/错误处理与文档不符 | **说清差在哪** + 一句最小修复方向 |
| ❌ **未实现** | 文档写了、代码里找不到痕迹 | 场景 3 下必须先做二次搜索排除误报 |
| ➕ **文档外实现** | 代码改了但文档没写 | 单列提醒补文档（场景 4 主要产出） |

### Step 5. 产出报告（落盘到 Q3 指定路径）

**报告设计原则**：
- **一眼见问题**：开头必须有"一眼看问题"段，一句话讲清本次最严重的 1~2 件事 + 状态统计表。
- **设计 vs 实现对比是核心**：主表一律用"设计要求 | 实际实现 | 差距/判定"三栏结构，禁止只有文字描述。
- **数字对齐数字、行为对齐行为**：偏差项要把文档里的具体数值/逻辑和代码里的具体数值/逻辑并排写，让读者一眼看到差在哪。
- **克制废话**：不写"本次改动涉及..."这种开场白；不写"分阶段小结"长段落；直接上表。
- **代码位置精确到行**：`<file>:<line>` 形式（如 `MainActivity:528`），而不是整文件。

## 红线

1. **能推就不问**：用户已经说清的信息不要再问；文档自带分阶段就自己裁决；只在真正不确定时才一次性合并 AskQuestion。
2. **推完要复述**：自动裁决结果必须用一段话打给用户看（标尺、文件清单、报告路径），让用户可当场喊停——这替代传统的强制确认闸门。
3. **只讲事实**：偏差说明必须指出"文档说 X，代码是 Y"这种具体差，不写"可能"、"也许"。
4. **最小修复方向**：一句话（如"把阈值 0.6 改回文档的 0.7"），不展开方案设计。
5. **范围不匹配立即止损**：如果发现 commit 主题与设计文档完全不在一条链路上（主题词、模块词、关键实体词都对不上），必须停下来让用户裁决，不硬跑。
6. **不混任务**：报告不讨论未来路线、不提"建议重构"，只回答"这次到底做到了什么"。
7. **报告语言跟随文档**：设计文档是中文就出中文报告，是英文就出英文报告，保持术语一致。

## 反模式（禁止）

- ❌ 用户已经给了的信息还再问一遍（如用户已明确指出文档路径，skill 还问"用哪份文档"）。
- ❌ 用户没给 Git 范围就自动 `git log -20` 猜。
- ❌ 文档自带分阶段标记，还去问用户"要审哪阶段"——应自行按 commit 时间裁决。
- ❌ 把整个 `git diff` 输出贴进报告上下文。
- ❌ 对"未实现"项下结论前不做二次搜索（可能在别的 commit 里）。
- ❌ 把 skill 自己的执行过程写进报告。
- ❌ 给"建议引入 xx 设计模式"这种超纲建议。
- ❌ 场景 1 下还去跑 Git（此时 Git 范围尚不存在）。
