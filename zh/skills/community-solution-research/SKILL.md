# 社区方案深度调研 (Community Solution Research)

## 核心原则

**"带着痛点去查，不要泛泛而搜"**

在外部社区寻找方案之前，必须先将内部业务痛点和工程约束翻译为技术锚点。外部调研的终极目的是解决我们自己的问题，而不是写学术综述。

## 执行流程

### Step 1: 内部约束提取

如果用户没有明确提供，必须先从项目文档、历史对话或代码中提取以下要素（这一步不可跳过）：

- **核心痛点**：当前系统哪里不行？（示例：意图路由延迟过高）
- **工程约束**：有什么客观限制？（示例：必须本地运行、禁止引入重型框架）
- **技术栈基建**：已经在用什么？（示例：已搭建向量数据库）

### Step 2: 搜索策略构建

禁止使用单纯的自然语言泛搜。必须使用多维度专业组合：

- **架构与标准类**：`[关键词] architecture` / `system design` / `best practices`
- **落地与避坑类**：`[关键词] in production` / `challenges` / `vs`
- **真实讨论类**：`[关键词] reddit` / `stack overflow`
- **开源生态类**：`[关键词] github` / `awesome [关键词]`

并发执行高维度查询，而不是串行慢搜。

### Step 3: 方案筛选与交叉验证

对每个候选方案，用 Step 1 的内部约束进行残酷过滤：

- 能解决核心痛点吗？
- 是否超出工程约束？（太重 / 太贵 / 需大量标注数据 / 维护成本过高？）
- 能无缝融入现有技术栈吗？还是需要推倒重来？

### Step 4: 结构化输出

必须按以下模板交付：

```markdown
## 社区调研报告：[主题]

### 1. 内部锚点
- **核心痛点**：...
- **工程约束**：...
- **现有基建**：...

### 2. 社区主流方案全景
- **方案 A**（如：Semantic Router）：[一句话原理]
- **方案 B**：[一句话原理]

### 3. 方案适配性对比

| 候选 | 解决痛点 | 违背约束 | 接入成本 | 推荐度 |
|---|---|---|---|---|
| A | 完美解决延迟 | 无 | 极低（复用现有基建）| ⭐⭐⭐⭐⭐ |
| B | 准确率高 | 需引入新重型依赖 | 高 | ⭐⭐ |

### 4. 落地建议与避坑指南
- **结论**：结合项目现状，强烈建议采用 [方案X]
- **社区踩坑预警**：根据社区反馈，实施时常遇到 [具体问题]，建议...
```

## 注意事项

- 坚决淘汰理论先进但严重违背系统约束的"屠龙术"方案
- 若搜索结果全是 SEO 营销文，立即在关键词后缀加 `github` / `issue` / 具体技术词后重试
