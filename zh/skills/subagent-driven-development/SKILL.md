# 子代理驱动开发

通过为每个任务或问题分派新子代理来创建和执行计划，在每个或批量任务后进行代码和输出审查。

**核心原则**：每任务新子代理 + 任务间审查 = 高质量、快速迭代。

**通过代理执行计划的优势**：
- 同一会话（无上下文切换）
- 每任务新子代理（无上下文污染）
- 每任务后代码审查（早期捕获问题）
- 更快迭代（任务间无人工干预）

---

## 支持的执行类型

### 顺序执行

当任务相互关联且需要按顺序执行时。

**何时使用**：
- 任务紧密耦合
- 任务应按顺序执行

### 并行执行

当有多个不相关任务或问题（不同文件、不同子系统、不同 bug）时。

**何时使用**：
- 任务基本独立
- 可以在所有任务完成后进行总体审查

---

## 顺序执行流程

### 1. 加载计划

读取计划文件，用 TodoWrite 创建所有任务。

### 2. 用子代理执行任务

对每个任务：

**分派新子代理**：

```markdown
Task 工具 (general-purpose):
    description: "实现任务 N: [任务名称]"
    prompt: |
        你正在实现 [计划文件] 中的任务 N。

        仔细阅读该任务。你的工作是：
        1. 准确实现任务指定的内容
        2. 编写测试（如果任务要求则遵循 TDD）
        3. 验证实现工作
        4. 提交你的工作
        5. 报告回来

        工作目录: [目录]

        报告: 你实现了什么，测试了什么，测试结果，修改的文件，任何问题
```

**子代理返回**工作摘要。

### 3. 审查子代理工作

**分派代码审查子代理**：

```markdown
Task 工具 (code-reviewer):
    WHAT_WAS_IMPLEMENTED: [来自子代理报告]
    PLAN_OR_REQUIREMENTS: [计划文件] 中的任务 N
    BASE_SHA: [任务前的提交]
    HEAD_SHA: [当前提交]
    DESCRIPTION: [任务摘要]
```

**代码审查返回**：优点、问题（关键/重要/次要）、评估

### 4. 应用审查反馈

**如果发现问题**：
- 立即修复关键问题
- 在下个任务前修复重要问题
- 记录次要问题

**如需要分派跟进子代理**：
```
"修复代码审查中的问题: [问题列表]"
```

### 5. 标记完成，下一任务

- 在 TodoWrite 中标记任务为已完成
- 移到下一任务
- 重复步骤 2-5

### 6. 最终审查

所有任务完成后，分派最终代码审查：
- 审查整个实现
- 检查所有计划需求是否满足
- 验证整体架构

## 并行执行流程

加载计划，批判性审查，批量执行任务，批次间报告供审查。

**核心原则**：批量执行带检查点供架构师审查。

### 步骤 1: 加载并审查计划

1. 读取计划文件
2. 批判性审查 - 识别任何问题或疑虑
3. 如有疑虑：开始前向人类伙伴提出
4. 如无疑虑：创建 TodoWrite 并继续

### 步骤 2: 执行批次

**默认：前 3 个任务**

对每个任务：
1. 标记为 in_progress
2. 准确遵循每个步骤
3. 运行指定的验证
4. 标记为 completed

### 步骤 3: 报告

批次完成时：
- 展示实现了什么
- 展示验证输出
- 说："准备好接收反馈。"

### 步骤 4: 继续

基于反馈：
- 如需要应用更改
- 执行下一批次
- 重复直到完成

---

## 并行调查流程

并行执行的特殊情况，当有多个不相关的失败可以独立调查时。

### 1. 识别独立领域

按问题分组失败：
- 文件 A 测试：工具审批流程
- 文件 B 测试：批量完成行为
- 文件 C 测试：中止功能

每个领域独立 - 修复工具审批不影响中止测试。

### 2. 创建聚焦的代理任务

每个代理获得：
- **具体范围**：一个测试文件或子系统
- **清晰目标**：让这些测试通过
- **约束**：不要修改其他代码
- **预期输出**：你发现和修复了什么的摘要

### 3. 并行分派

```python
# 在 Claude Code / AI 环境中
Task("修复 agent-tool-abort.test.ts 失败")
Task("修复 batch-completion-behavior.test.ts 失败")
Task("修复 tool-approval-race-conditions.test.ts 失败")
# 三个并发运行
```

### 4. 审查和集成

代理返回时：
- 阅读每个摘要
- 验证修复不冲突
- 运行完整测试套件
- 集成所有更改

## 代理提示结构

好的代理提示是：

1. **聚焦** - 一个清晰的问题领域
2. **自包含** - 理解问题所需的所有上下文
3. **具体的输出** - 代理应该返回什么？

**好例子**：

```markdown
修复 src/agents/agent-tool-abort.test.ts 中的 3 个失败测试：

1. "should abort tool with partial output capture" - 期望消息中有 'interrupted at'
2. "should handle mixed completed and aborted tools" - 快速工具被中止而非完成
3. "should properly track pendingToolCount" - 期望 3 个结果但得到 0

这些是时序/竞态条件问题。你的任务：

1. 阅读测试文件并理解每个测试验证什么
2. 识别根本原因 - 时序问题还是实际 bug？
3. 修复方式：
   - 用基于事件的等待替换任意超时
   - 如果发现 bug 则修复中止实现
   - 如果测试的行为已改变则调整测试期望

不要只是增加超时 - 找到真正的问题。

返回：你发现了什么和修复了什么的摘要。
```

---

## 常见错误

| 错误 | 正确 |
|------|------|
| **太宽泛**："修复所有测试" | **具体**："修复 agent-tool-abort.test.ts" |
| **无上下文**："修复竞态条件" | **有上下文**：粘贴错误消息和测试名称 |
| **无约束**：代理可能重构所有 | **有约束**："不要修改生产代码" |
| **模糊输出**："修复它" | **具体输出**："返回根本原因和更改的摘要" |

---

## 何时不使用

- **相关失败**：修复一个可能修复其他 - 先一起调查
- **需要完整上下文**：理解需要看到整个系统
- **探索性调试**：你还不知道什么坏了
- **共享状态**：代理会干扰（编辑相同文件、使用相同资源）

## 真实示例

**场景**：主要重构后 3 个文件中有 6 个测试失败

**失败**：
- agent-tool-abort.test.ts：3 个失败（时序问题）
- batch-completion-behavior.test.ts：2 个失败（工具未执行）
- tool-approval-race-conditions.test.ts：1 个失败（执行计数 = 0）

**决策**：独立领域 - 中止逻辑与批量完成与竞态条件分开

**分派**：
```
代理 1 → 修复 agent-tool-abort.test.ts
代理 2 → 修复 batch-completion-behavior.test.ts
代理 3 → 修复 tool-approval-race-conditions.test.ts
```

**结果**：
- 代理 1：用基于事件的等待替换超时
- 代理 2：修复事件结构 bug（threadId 在错误位置）
- 代理 3：添加等待异步工具执行完成

**集成**：所有修复独立，无冲突，完整套件绿色

**节省时间**：3 个问题并行解决 vs 顺序

## 验证

代理返回后：

1. **阅读每个摘要** - 理解什么改变了
2. **检查冲突** - 代理是否编辑了相同代码？
3. **运行完整套件** - 验证所有修复一起工作
4. **抽查** - 代理可能犯系统性错误

---

## 红旗

**绝不**：
- 跳过任务间代码审查
- 带着未修复的关键问题继续
- 并行分派多个实现子代理（冲突）
- 不读计划任务就实现

**如果子代理任务失败**：
- 用具体指令分派修复子代理
- 不要尝试手动修复（上下文污染）

---

## 何时停下并寻求帮助

**立即停止执行当**：
- 批次中遇到阻塞（缺少依赖、测试失败、指令不清晰）
- 计划有阻止开始的关键缺口
- 你不理解一个指令
- 验证反复失败

**寻求澄清而非猜测。**
