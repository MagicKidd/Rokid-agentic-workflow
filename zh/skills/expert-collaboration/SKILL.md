# 专家协作（Expert Collaboration）

## 适用场景

- 任务需要多领域专家配合
- 无重大分歧，只有职责和视角差异
- 需要渐进收敛产出可执行结果
- 复杂任务的分工执行

## 协作规则

1. 每位专家贡献各自领域的工作或建议
2. 专家之间相互回应、渐进收敛
3. **协作模式需手动指定**，不会自动切换为辩论
4. 限制影响范围，优先小步可验证

## 协作流程

### 启动格式

```
启动 **专家协作**
角色: [角色1], [角色2], ...
任务: [需要完成的任务]
```

### 协作轮次结构

**每轮每位专家：**
1. **推理（Reasoning）**：分析当前状态和下一步
2. **行动（Acting）**：执行或建议具体操作
3. **观察（Observing）**：评估结果，为下轮准备

### 输出格式

```markdown
## 第 N 轮

### [角色1]
**推理**: 当前状态分析...
**行动**: 我负责...
**观察**: 完成情况...

### [角色2]
**推理**: 基于[角色1]的产出...
**行动**: 我补充...
**观察**: 整体进度...
```

## 协作结束

至少三轮，直到所有专家达成一致：

```markdown
## 协作成果

### 各专家贡献
- [角色1]: 完成了...
- [角色2]: 完成了...

### 最终产出
[可执行的结果/方案/代码]

### 验证清单
- [ ] 检查点1
- [ ] 检查点2
```

## 示例

**用户**: 用专家协作帮我设计一个用户认证模块

**响应**:

```
启动 **专家协作**
角色: System Architect, Full-stack Developer, Test/QA Expert
任务: 设计用户认证模块

## 第 1 轮

### System Architect
**推理**: 认证模块需要考虑安全性、可扩展性、与现有系统集成
**行动**: 
- 选定 JWT + Refresh Token 方案
- 定义接口规范：/auth/login, /auth/refresh, /auth/logout
**观察**: 架构层面完成，需要开发实现细节

### Full-stack Developer
**推理**: 基于架构师的方案，需要实现具体代码
**行动**: 
- 实现 JWT 签发和验证中间件
- 设计 Token 存储策略（Redis）
**观察**: 核心功能完成，需要测试覆盖

### Test/QA Expert
**推理**: 认证模块是安全关键，需要全面测试
**行动**:
- 设计测试用例：正常登录、Token过期、并发刷新
- 安全测试：SQL注入、暴力破解防护
**观察**: 测试计划完成

...（继续2-3轮完善）...

## 协作成果
### 最终产出
完整的认证模块设计文档 + 核心代码 + 测试用例
```

## 与专家辩论的区别

| 维度 | 专家协作 | 专家辩论 |
|------|----------|----------|
| 前提 | 无重大分歧 | 存在分歧 |
| 目标 | 分工完成任务 | 统一策略 |
| 产出 | 可执行结果 | 决策共识 |
| 过程 | 各自贡献、渐进收敛 | 相互质疑、推动收敛 |
