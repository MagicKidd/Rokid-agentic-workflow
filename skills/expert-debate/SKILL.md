# 专家辩论（Expert Debate）

## 适用场景

- 两位专家对同一问题持相反观点
- 技术方案路径分歧
- 模型边界或职责不清晰
- 需要统一策略或明确方向

## 辩论规则

1. **专家不执行任务**，只提供判断和理由
2. 每轮相互回应对方观点，推动收敛
3. **辩论期间不输出代码或设计产物**，只总结共识与分歧
4. 辩论结束后才进入执行阶段

## 辩论流程

### 启动格式

```
启动 **专家辩论**
专家: [角色1], [角色2]
议题: [需要辩论的问题]
```

### 三轮辩论结构

**每轮包含：**
1. **判断（Judgment）**：对议题的立场
2. **推理（Reasoning）**：支撑立场的论据
3. **回应（Rebuttal/Support）**：对对方观点的反驳或支持

### 输出格式

```markdown
## 第 N 轮

### [角色1]
**判断**: ...
**推理**: ...
**对[角色2]的回应**: ...

### [角色2]
**判断**: ...
**推理**: ...
**对[角色1]的回应**: ...
```

## 辩论结束

第三轮后，输出：

```markdown
## 辩论总结

### 达成共识
- 共识点1
- 共识点2

### 仍有分歧
- 分歧点1（需人工决策）

### 建议策略
[统一策略或推荐方案]
```

## 示例

**用户**: 用专家辩论分析一下：这个模块应该用微服务还是单体架构？

**响应**:

```
启动 **专家辩论**
专家: System Architect, Full-stack Developer
议题: 微服务 vs 单体架构的选择

## 第 1 轮

### System Architect
**判断**: 推荐微服务架构
**推理**: 
- 模块独立部署，降低耦合
- 团队可并行开发
- 便于未来扩展
**对 Full-stack Developer 的回应**: 待对方发言

### Full-stack Developer
**判断**: 推荐单体架构
**推理**:
- 当前团队规模小，微服务增加运维复杂度
- 初期业务不确定，单体便于快速迭代
- 避免过早优化
**对 System Architect 的回应**: 微服务的优势在团队规模扩大后才显现

...（继续2-3轮）...

## 辩论总结
### 达成共识
- 初期采用模块化单体，为未来拆分预留接口

### 建议策略
先单体后微服务，设定拆分触发条件（如团队>10人或模块QPS>1000）
```
