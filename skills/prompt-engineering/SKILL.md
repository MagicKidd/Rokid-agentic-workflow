# 提示工程模式

高级提示工程技术，最大化 LLM 性能、可靠性和可控性。

## 核心能力

### 1. Few-Shot 学习

通过展示示例而非解释规则来教导模型。包含 2-5 个输入-输出对来演示期望行为。

**何时使用**：需要一致格式、特定推理模式或处理边缘情况时。

**示例**：

```markdown
从支持工单中提取关键信息：

Input: "登录不工作，一直报错403"
Output: {"issue": "authentication", "error_code": "403", "priority": "high"}

Input: "功能请求：在设置中添加暗黑模式"
Output: {"issue": "feature_request", "error_code": null, "priority": "low"}

现在处理: "无法上传超过10MB的文件，超时了"
```

### 2. 思维链（Chain-of-Thought）提示

在最终答案前请求逐步推理。

**零样本**：添加 "让我们一步步思考"
**少样本**：包含示例推理轨迹

**示例**：

```markdown
分析此 bug 报告并确定根本原因。

一步步思考：
1. 期望行为是什么？
2. 实际行为是什么？
3. 最近有什么变化可能导致这个问题？
4. 涉及哪些组件？
5. 最可能的根本原因是什么？

Bug: "昨天缓存更新部署后，用户无法保存草稿"
```

### 3. 提示优化流程

通过测试和迭代系统性改进提示词。

```markdown
版本 1（简单）: "总结这篇文章"
→ 结果: 长度不一致，遗漏关键点

版本 2（添加约束）: "用3个要点总结"
→ 结果: 结构更好，但仍遗漏细微差别

版本 3（添加推理）: "找出3个主要发现，然后分别总结"
→ 结果: 一致、准确、捕获关键信息
```

### 4. 模板系统

构建带变量、条件段和模块化组件的可复用提示结构。

```python
# 可复用的代码审查模板
template = """
审查这段 {language} 代码的 {focus_area}。

代码:
{code_block}

提供以下方面的反馈:
{checklist}
"""

# 使用
prompt = template.format(
    language="Python",
    focus_area="安全漏洞",
    code_block=user_code,
    checklist="1. SQL注入\n2. XSS风险\n3. 认证问题"
)
```

### 5. 系统提示设计

设置持续整个对话的全局行为和约束。

```markdown
System: 你是一位专注于 API 设计的高级后端工程师。

规则:
- 始终考虑可扩展性和性能
- 默认建议 RESTful 模式
- 立即标记安全问题
- 用 Python 提供代码示例
- 使用早期返回模式

响应格式:
1. 分析
2. 建议
3. 代码示例
4. 权衡
```

## 关键模式

### 渐进披露

从简单提示开始，仅在需要时增加复杂度：

1. **级别1**: 直接指令 → "总结这篇文章"
2. **级别2**: 添加约束 → "用3个要点总结，聚焦关键发现"
3. **级别3**: 添加推理 → "阅读文章，识别主要发现，然后用3个要点总结"
4. **级别4**: 添加示例 → 包含2-3个输入-输出对

### 指令层次

```
[系统上下文] → [任务指令] → [示例] → [输入数据] → [输出格式]
```

### 错误恢复

构建能优雅处理失败的提示：
- 包含回退指令
- 请求置信度分数
- 不确定时请求替代解释
- 指定如何表示缺失信息

## 上下文窗口管理

**核心原则**：上下文窗口是公共资源

你的提示与以下内容共享上下文窗口：
- 系统提示
- 对话历史
- 其他命令、技能、钩子、元数据
- 实际请求

**默认假设**：模型已经很聪明

只添加模型不知道的上下文。挑战每条信息：
- "模型真的需要这个解释吗？"
- "我能假设模型知道这个吗？"
- "这段话值得消耗 token 吗？"

**好例子（约50 tokens）**：
```markdown
## 提取 PDF 文本
使用 pdfplumber：
\`\`\`python
import pdfplumber
with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
\`\`\`
```

**坏例子（约150 tokens）**：
```markdown
## 提取 PDF 文本
PDF（便携文档格式）是一种常见的文件格式，包含文本、图像和其他内容。
要从 PDF 提取文本，你需要使用一个库。有很多库可用于 PDF 处理，
但我们推荐 pdfplumber 因为它易于使用且能处理大多数情况...
```

## 自由度设置

根据任务脆弱性和可变性匹配具体程度。

| 自由度 | 何时使用 | 示例 |
|--------|----------|------|
| **高**（文本指令） | 多种方法有效，取决于上下文 | 代码审查指南 |
| **中**（伪代码/模板） | 有首选模式但允许变化 | 报告生成 |
| **低**（具体脚本） | 操作脆弱，一致性关键 | 数据库迁移 |

## 说服原则（Agent 提示设计）

LLM 对与人类相同的说服原则有响应。

### 1. 权威（Authority）

- 命令式语言："必须"、"绝不"、"始终"
- 不可协商框架："无例外"
- 消除决策疲劳和合理化

```markdown
✅ 在测试前写代码？删除它。重新开始。无例外。
❌ 如果可行的话考虑先写测试。
```

### 2. 承诺（Commitment）

- 要求宣布："宣布技能使用"
- 强制明确选择："选择 A、B 或 C"
- 使用跟踪：TodoWrite 用于检查清单

```markdown
✅ 当你找到一个技能，你必须宣布："我正在使用 [技能名称]"
❌ 考虑让你的伙伴知道你在使用哪个技能。
```

### 3. 社会证明（Social Proof）

- 通用模式："每次"、"始终"
- 失败模式："没有Y的X = 失败"

```markdown
✅ 没有 TodoWrite 跟踪的检查清单 = 步骤会被跳过。每次如此。
❌ 有些人发现 TodoWrite 对检查清单有帮助。
```

### 4. 统一（Unity）

- 协作语言："我们的代码库"、"我们是同事"
- 共同目标："我们都想要质量"

```markdown
✅ 我们是一起工作的同事。我需要你诚实的技术判断。
❌ 如果我错了你应该告诉我。
```

## 最佳实践清单

1. **具体**：模糊提示产生不一致结果
2. **展示而非讲述**：示例比描述更有效
3. **广泛测试**：在多样化、代表性输入上评估
4. **快速迭代**：小改变可能有大影响
5. **监控性能**：在生产中跟踪指标
6. **版本控制**：像代码一样对待提示词
7. **记录意图**：解释为什么这样构建提示

## 常见陷阱

- **过度工程**：在尝试简单方案前就构建复杂提示
- **示例污染**：使用与目标任务不匹配的示例
- **上下文溢出**：过多示例超出 token 限制
- **模糊指令**：留有多种解释的空间
- **忽略边缘情况**：不在异常或边界输入上测试
